# Query API with Document Highlighting

## Context

**Goal:** Enable the frontend to highlight relevant sections of documents based on retrieval results from the query API.

**Current state:** The backend implements a PageIndex-like tree-structured indexing and iterative retrieval system. The frontend has `react-pdf-highlighter` integrated but uses hardcoded highlights and loads from a static `/example.pdf`. The query API returns citations, but they are LLM-generated (unreliable) rather than programmatically tracked from the retrieval path.

**Greenfield assumption:** This is a greenfield API. We can replace the current query response contract if a cleaner design emerges.

---

## Our Implementation vs PageIndex

PageIndex is an open-source, vectorless RAG framework by VectifyAI that builds hierarchical tree indices over documents and uses LLM reasoning to navigate them for retrieval. It achieved 98.7% accuracy on FinanceBench.

| Aspect | Mirage (current) | PageIndex |
|---|---|---|
| **Index structure** | Hierarchical tree: `node_id`, `start_page`, `end_page`, `summary`, `children` | Nearly identical tree with same fields |
| **TOC detection** | LLM analyzes first 20 pages (`min(20, len(pages))`) | LLM-based, configurable page count |
| **Structure inference** | LLM groups pages when no TOC found | Similar fallback |
| **Retrieval** | Iterative branch selection + sufficiency check (up to 5 iterations) | Iterative 5-step reasoning loop (read TOC, select, extract, evaluate, answer/loop) |
| **Citation tracking** | LLM generates citations in the answer step — non-deterministic | Deterministic: returns `retrieved_nodes` (node IDs + titles) and `relevant_contents` (page index + text excerpts) as byproducts of traversal |
| **Tree API** | Tree stored as `tree.json` per document, not exposed via API | `GET /doc/{doc_id}/?type=tree` returns full tree with summaries, page indices, node IDs |
| **Cross-references** | Not handled | LLM can navigate to referenced sections |

**Key insight:** Mirage is already 90% aligned with PageIndex. Two critical gaps:

1. **Citations are non-deterministic.** They are generated by the LLM during answer synthesis (`generateAnswer` in `retrieval.go`), not tracked programmatically. The retrieval loop already visits specific nodes and fetches specific pages — it just discards this provenance before producing the final result.
2. **Tree is not exposed.** PageIndex serves the tree via API, letting frontends show document outlines, resolve `node_id` → section title + page range, and enable section-aware navigation. Mirage already stores `tree.json` but has no endpoint to serve it.

Sources:
- https://docs.pageindex.ai/endpoints
- https://docs.pageindex.ai/tutorials/llm-tree-search-rag
- https://github.com/VectifyAI/PageIndex

## Gap Analysis

1. **No evidence payload in API** — Frontend receives citation IDs but not highlightable text or page ranges from traversed nodes.
2. **Citation grounding is weak** — Final citation list is LLM-produced and not validated against the nodes/pages actually traversed.
3. **No PDF serving endpoint** — Frontend loads a static `/example.pdf`, not the uploaded document.
4. **No tree endpoint** — Document structure is stored but not accessible to the frontend.
5. **Page-number consistency risk** — Internal logic is 0-based; `react-pdf-highlighter` is 1-based.
6. **Frontend query→highlight path not wired** — Current PDF highlighting is mock/static and independent of query responses.

---

## Proposals

### Proposal A: Deterministic Evidence with Tree API

**Core idea:** Track which tree nodes and pages are actually visited during retrieval (instead of asking the LLM to produce citations), expose the tree structure, and serve PDFs from the backend.

Replace the current `citations[]` with an `evidence[]` model:

```json
{
  "answer": "string",
  "evidence": [
    {
      "document_id": "string",
      "document_name": "string",
      "node_id": "0007",
      "node_title": "Financial Results",
      "page_start": 5,
      "page_end": 8,
      "snippet": "relevant markdown content from the page(s)"
    }
  ]
}
```

All `page_*` fields are **1-indexed** to match PDF viewer conventions.

**Backend changes:**
- `srv/internal/service/retrieval.go` — Build evidence from visited leaf nodes during the iteration loop. The data is already available: `visitedNodes`, `cand.StartPage`/`EndPage`, `cand.Title`, and the fetched page markdown. Stop asking the LLM to generate citations.
- `srv/internal/domain/query.go` — Replace `Citation` with `Evidence` struct carrying `node_id`, `node_title`, `page_start`, `page_end`, `snippet`.
- `srv/internal/api/documents.go` — Add two endpoints:
  - `GET /api/v1/documents/{id}/pdf` — serve the source PDF binary (storage already has `OpenPDF`)
  - `GET /api/v1/documents/{id}/tree` — return the document's tree index (storage already has `GetTree`)
- `srv/internal/api/dto.go` — Update DTOs for evidence and tree responses.

**Frontend changes:**
- `web/src/lib/api.ts` — Add `getDocumentPdfUrl(id)` and `getDocumentTree(id)`.
- `web/src/components/PdfViewer.tsx` — Load PDF from backend by document ID. Convert evidence page ranges into page-level highlights.
- Use tree data to display section titles alongside highlights (e.g., "Financial Results, pages 5-8").

**Trade-offs:**
- (+) Minimal backend changes — reuses data already computed during retrieval
- (+) Citations become deterministic, grounded in actual traversal
- (+) Tree endpoint is trivial — just serving existing `tree.json`
- (+) Section-aware highlights give meaningful context
- (+) `snippet` field enables future text-level matching on the frontend
- (-) Page-level highlighting only — does not pinpoint specific sentences within a page

---

### Proposal B: Evidence with Text Anchors

**Core idea:** Additive to Proposal A. Add character-offset anchors to evidence items, enabling precise sub-page highlighting.

```json
{
  "anchor": {
    "kind": "text",
    "char_start": 421,
    "char_end": 509
  }
}
```

The anchor offsets are relative to the page's markdown text (stored in `pages.json`).

**Backend changes (additive to A):**
- During evidence collection, compute character offsets of the relevant snippet within the page markdown.
- Store normalized page text alongside markdown if OCR output changes require stability.

**Frontend changes (additive to A):**
- Use `react-pdf-highlighter`'s text layer to match snippet text within the rendered PDF page and create precise text-level highlights.

**Trade-offs:**
- (+) More precise highlights — users see exactly which sentences matter
- (+) Better UX for long pages with mixed relevant/irrelevant content
- (+) `anchor.kind` is extensible (can add `geometry` later)
- (-) Text matching can fail due to OCR artifacts or markdown↔PDF text divergence
- (-) Requires offset bookkeeping during retrieval
- (-) More complex than page-level highlighting

---

### Proposal C: Retrieval Reasoning Trace

**Core idea:** Additive to Proposal A. Record and expose the retrieval reasoning path.

```json
{
  "answer": "string",
  "evidence": [...],
  "reasoning_trace": [
    {
      "action": "select_branches",
      "node_ids": ["0003", "0007"],
      "reasoning": "These sections discuss financial performance...",
      "sufficient": false
    }
  ]
}
```

**Backend changes (additive to A):**
- `srv/internal/service/retrieval.go` — Record each iteration's branch selection reasoning and sufficiency check. The `ReasoningStep` type already exists in `domain/query.go` but isn't populated.

**Frontend changes (additive to A):**
- Tree visualization component showing document structure with visited/selected nodes highlighted.
- Collapsible reasoning panel showing step-by-step navigation.

**Trade-offs:**
- (+) Full transparency into retrieval decisions
- (+) Debugging value — reveals when retrieval goes wrong
- (+) Enables future interactive tree-guided search
- (-) Significant frontend complexity
- (-) Most users don't need to see the reasoning trace

---

## Recommendation

**Start with Proposal A.** It delivers the core value — deterministic, section-aware highlighting — with minimal changes. The tree endpoint is nearly free (data already exists), and evidence grounded in actual traversal is fundamentally more reliable than LLM-generated citations.

**Proposal B** (text anchors) can be layered on if page-level highlighting feels too coarse.

**Proposal C** (reasoning trace) is valuable for debugging and power users but should wait until the core query→highlight loop is proven.

---

## Iterative Implementation Plan

### Iteration 1: Ship the core loop

- Replace `citations[]` with `evidence[]` grounded in traversal.
- Add `GET /documents/{id}/pdf` and `GET /documents/{id}/tree` endpoints.
- Wire frontend: query → evidence → PDF viewer with page-level highlights.
- All page numbers 1-indexed in the API contract.

### Iteration 2: Stabilize

- Improve snippet quality (trim to most relevant passage within multi-page nodes).
- Add retrieval trace behind a debug query parameter for tuning.
- Add tests for: 1-indexed page contract, evidence grounding to traversed nodes.

### Iteration 3: Optional precision upgrade

- Add text anchors (Proposal B) once page-level UX and costs are validated.
- Add geometry anchors if OCR provider supports bounding box output.

## Extensibility Notes

- Keep orchestration in `service.Retrieval`; isolate evidence generation behind a clear boundary (future candidate: `EvidenceExtractor` interface).
- Keep storage additive — optional files like `page_positions.json` can be added later without migrating existing documents.
- Keep `anchor.kind` extensible (`text` now, `geometry` later) so the frontend can handle both.

## Future Considerations

- **Streaming query responses** — Retrieval can take 10-30s. Stream partial results (navigation progress, intermediate evidence) via SSE.
- **Multi-document highlighting** — Current UI shows one PDF at a time. Consider split-view or document tabs for cross-document queries.
- **Persisted highlights/annotations** — Let users save, share, and annotate highlights.
- **Interactive tree-guided search** — Let users click tree nodes to pre-scope queries to specific sections.
- **Evidence ranking and confidence scoring** — Rank evidence items by relevance.
- **Caching retrieval results** — Avoid re-running expensive LLM queries for repeated questions.
- **Vector search hybrid** — Add embedding-based pre-filtering for large document collections before tree navigation.
