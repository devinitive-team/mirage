# PageIndex: Technical Overview

> Vectorless, Reasoning-Based RAG Framework

## 1. What Is PageIndex?

PageIndex (by VectifyAI) is a document retrieval framework that replaces vector
embeddings and similarity search with **hierarchical tree indexing** and
**LLM-driven reasoning**. Instead of asking "what text is semantically similar
to the query?", it asks "where in this document would the answer logically
live?"

Core thesis: **similarity ≠ relevance**. Semantic closeness (as measured by
vector distance) does not guarantee that a chunk actually answers the question.

- Open-source repo: https://github.com/VectifyAI/PageIndex
- Docs: https://docs.pageindex.ai
- Cloud service: https://chat.pageindex.ai
- Benchmark: **98.7% accuracy on FinanceBench** (via Mafin 2.5)

---

## 2. Why Not Vector Search?

PageIndex explicitly targets five failure modes of traditional vector-based RAG:

| Problem | Vector RAG Behavior | PageIndex Approach |
|---|---|---|
| **Query-knowledge mismatch** | Embeds the query and hopes nearby chunks answer it | LLM reasons about document structure to infer where the answer lives |
| **Similarity ≠ relevance** | Returns chunks that _look like_ the query but may not answer it | Navigates to sections that _logically contain_ the answer |
| **Hard chunking** | Fixed-size chunks destroy context, split tables, break cross-references | Preserves natural document hierarchy (chapters, sections, paragraphs) |
| **No chat history awareness** | Each query is an isolated embedding lookup | Multi-turn reasoning leverages conversation context |
| **Cross-reference failure** | "See Appendix G" is meaningless to a cosine similarity score | LLM follows references by navigating the tree |

## 3. Architecture

### 3.1 Two-Phase Pipeline

```
┌─────────────────┐         ┌─────────────────────┐
│  Phase 1: Index │         │  Phase 2: Retrieval  │
│  (offline)      │         │  (per query)         │
│                 │         │                      │
│  PDF/Markdown   │         │  Query arrives       │
│       │         │         │       │              │
│       ▼         │         │       ▼              │
│  OCR + Layout   │         │  LLM reads tree      │
│  Preservation   │         │       │              │
│       │         │         │       ▼              │
│       ▼         │         │  Selects branch      │
│  Tree Index     │         │       │              │
│  Generation     │         │       ▼              │
│       │         │         │  Extracts content    │
│       ▼         │         │       │              │
│  JSON tree      │─────────│       ▼              │
│  with summaries │         │  Sufficiency check   │
│                 │         │       │              │
│                 │         │  ┌────┴────┐         │
│                 │         │  │ enough? │         │
│                 │         │  └────┬────┘         │
│                 │         │   no  │  yes         │
│                 │         │   │   │              │
│                 │         │   │   ▼              │
│                 │         │   │  Answer          │
│                 │         │   │  + citations     │
│                 │         │   │                  │
│                 │         │   └──→ loop back     │
└─────────────────┘         └─────────────────────┘
```

### 3.2 The Tree Index (Data Structure)

A document is transformed into a recursive JSON tree. Each node represents a
logical section of the document:

```json
{
  "title": "Annual Report 2024",
  "node_id": "0001",
  "start_index": 1,
  "end_index": 120,
  "summary": "Full annual report covering financials, operations, and outlook.",
  "nodes": [
    {
      "title": "Financial Statements",
      "node_id": "0002",
      "start_index": 15,
      "end_index": 45,
      "summary": "Consolidated income statement, balance sheet, and cash flows.",
      "nodes": [
        {
          "title": "Income Statement",
          "node_id": "0003",
          "start_index": 15,
          "end_index": 22,
          "summary": "Revenue of $4.2B, net income of $890M.",
          "nodes": []
        }
      ]
    }
  ]
}
```

**Node properties:**

| Field | Type | Description |
|---|---|---|
| `title` | string | Section heading |
| `node_id` | string | Unique identifier (e.g. `"0006"`) |
| `start_index` | int | First page of this section |
| `end_index` | int | Last page of this section |
| `summary` | string | LLM-generated summary of section content |
| `nodes` | array | Child sections (recursive) |
| `text` | string | Full text content of the node (when requested) |
| `page_index` | int | Source page number |

The tree mirrors a table of contents: chapter → section → subsection → paragraph.
Summaries are generated by the LLM during indexing, giving the retrieval LLM
enough context to decide which branches to explore without reading the full text.

### 3.3 LLM Tree Search (Retrieval Algorithm)

When a query arrives, the LLM performs an iterative reasoning loop:

1. **Read the tree** — the LLM receives the top-level nodes (titles + summaries)
   as context
2. **Select a branch** — the LLM reasons about which node(s) are most likely to
   contain the answer and returns their `node_id`s
3. **Drill down** — the selected node's children are presented; the LLM repeats
   the selection at the next level
4. **Extract content** — once at a leaf or sufficiently specific node, the
   system extracts the actual text from those pages
5. **Sufficiency check** — the LLM evaluates whether the extracted information
   answers the query
6. **Loop or answer** — if insufficient, loop back to step 1 and explore
   different branches; if sufficient, generate the final answer with exact
   page citations

This is fundamentally different from vector search's single-pass "top-K nearest
neighbors" approach. The LLM can:

- Follow cross-references ("see Appendix G") by navigating the tree
- Aggregate information from multiple sections
- Understand that section 3.2 _depends on_ definitions in section 1.1
- Abandon a dead-end branch and try a different one

### 3.4 Vision-Based Retrieval

For documents where layout matters (financial tables, charts, diagrams),
PageIndex supports a vision pipeline that skips OCR entirely. The LLM reasons
over the tree structure but retrieves **page images** instead of extracted text,
allowing a vision-language model to "see" the original document layout.

### 3.5 OCR Pipeline

For text-based retrieval, the OCR system processes documents as cohesive units
using vision-language models (not traditional character-level OCR). It preserves:

- Hierarchical headings
- Tables and their structure
- Bullet lists
- Cross-page continuity (a paragraph split across pages stays unified)

Output formats:
- `"page"` — per-page markdown + base64 images
- `"node"` — hierarchical node structure matching the tree
- `"raw"` — concatenated markdown string

## 4. API Reference

### 4.1 Authentication

All endpoints require an API key from https://dash.pageindex.ai/api-keys,
passed in the `api_key` header.

### 4.2 Python SDK

```bash
pip install -U pageindex
```

```python
from pageindex import PageIndexClient

pi_client = PageIndexClient(api_key="YOUR_API_KEY")
```

### 4.3 Document Lifecycle

#### Submit a document

```python
result = pi_client.submit_document("./report.pdf")
doc_id = result["doc_id"]  # e.g. "pi-abc123def456"
```

With MCP mode (makes document available via MCP protocol):
```python
result = pi_client.submit_document("./report.pdf", mode="mcp")
```

#### Check processing status

```python
status = pi_client.get_document(doc_id)["status"]
# "processing" | "completed"
```

#### Get the tree index

```python
tree = pi_client.get_tree(doc_id)["result"]
# With summaries:
tree = pi_client.get_tree(doc_id, node_summary=True)["result"]
```

#### Get OCR results

```python
ocr = pi_client.get_ocr(doc_id, format="page")  # or "node", "raw"
```

#### Delete a document

```python
pi_client.delete_document(doc_id)
```

### 4.4 REST Endpoints

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/doc/` | Submit PDF (multipart/form-data: `file`, optional `mode`) |
| `GET` | `/doc/{doc_id}/` | Get tree or OCR results (`type=tree\|ocr`, `format`, `summary`) |
| `GET` | `/doc/{doc_id}/metadata` | Document metadata (name, status, page count, etc.) |
| `GET` | `/docs` | List documents (`limit`, `offset`) |
| `DELETE` | `/doc/{doc_id}/` | Delete document |
| `POST` | `/chat/completions` | Chat API (beta) |
| `POST` | `/markdown/` | Convert markdown to tree index |
| `POST` | `/retrieval/` | Submit retrieval query (legacy) |
| `GET` | `/retrieval/{id}/` | Get retrieval results (legacy) |

Base URL: `https://api.pageindex.ai`

### 4.5 Chat API (Beta)

Query one or multiple documents with reasoning-based retrieval:

```python
response = pi_client.chat_completions(
    messages=[{"role": "user", "content": "What was Q3 revenue?"}],
    doc_id="pi-abc123def456",       # single doc
    # doc_id=["pi-...", "pi-..."],  # or multiple docs
    stream=False,                    # or True for SSE
    temperature=0.0,
    enable_citations=True            # inline <doc=file.pdf;page=1> refs
)

answer = response["choices"][0]["message"]["content"]
```

Streaming responses use Server-Sent Events (SSE) with `data:`-prefixed JSON
chunks. The stream includes `block_metadata` events for tracing the retrieval
steps:
- `text_block_start` / `text_block_stop`
- `mcp_tool_use_start` / `mcp_tool_use_stop`
- `mcp_tool_result_start` / `mcp_tool_result_stop`

### 4.6 Retrieval API (Legacy)

Lower-level access to the tree search step:

```python
# Submit query
result = pi_client.retrieval(doc_id=doc_id, query="...", thinking=True)
retrieval_id = result["retrieval_id"]

# Poll for results
result = pi_client.get_retrieval(retrieval_id)
# Returns: retrieved_nodes[] with page_index and content
```

### 4.7 Markdown-to-Tree

Convert markdown documents to tree indexes using heading hierarchy (`#`, `##`,
`###`, etc.):

```
POST /markdown/
Content-Type: multipart/form-data

file: report.md
if_add_node_id: "yes"
if_add_node_summary: "yes"
if_add_node_text: "yes"
if_add_doc_description: "yes"
```

Response: `{success, doc_name, structure}` with hierarchical nodes containing
`title`, `node_id`, `summary`, `line_num`.

## 5. Self-Hosted Indexing

The open-source repository allows local tree index generation:

```bash
git clone https://github.com/VectifyAI/PageIndex.git
cd PageIndex
pip3 install --upgrade -r requirements.txt
```

Create `.env`:
```
CHATGPT_API_KEY=your_openai_key_here
```

Run:
```bash
python3 run_pageindex.py --pdf_path /path/to/document.pdf
# or
python3 run_pageindex.py --md_path /path/to/document.md
```

**Configuration flags:**

| Flag | Default | Description |
|---|---|---|
| `--model` | `gpt-4o-2024-11-20` | LLM used for tree generation |
| `--toc-check-pages` | `20` | Pages scanned for table-of-contents detection |
| `--max-pages-per-node` | `10` | Max pages a single node can span |
| `--max-tokens-per-node` | `20000` | Token limit per node |
| `--if-add-node-id` | `yes` | Include node IDs in output |
| `--if-add-node-summary` | `yes` | Generate LLM summaries per node |
| `--if-add-doc-description` | `yes` | Generate document-level description |

Output: JSON tree structure written to disk.

## 6. MCP Integration

PageIndex exposes an MCP (Model Context Protocol) server, enabling any
MCP-compatible LLM client (Claude, Cursor, custom agents) to use PageIndex as a
tool for document retrieval.

**Cloud (API key auth):**
```json
{
  "mcpServers": {
    "pageindex": {
      "type": "http",
      "url": "https://api.pageindex.ai/mcp",
      "headers": {
        "Authorization": "Bearer YOUR_API_KEY"
      }
    }
  }
}
```

**Cloud (OAuth, for chat.pageindex.ai users):**
```json
{
  "mcpServers": {
    "pageindex": {
      "type": "http",
      "url": "https://chat.pageindex.ai/mcp"
    }
  }
}
```

**Local (Node.js ≥ 18):**
```json
{
  "mcpServers": {
    "pageindex": {
      "command": "npx",
      "args": ["-y", "@pageindex/mcp"]
    }
  }
}
```

Compatible with: Claude, Cursor, Claude Agent SDK, Vercel AI SDK, OpenAI Agents
SDK, LangChain, and any MCP-compatible client.

## 7. When to Use PageIndex vs Vector Search

| Scenario | Best Fit |
|---|---|
| Single long document, repeated queries | **PageIndex** — amortize index cost |
| Structured documents (contracts, reports, manuals) | **PageIndex** — structure matters |
| Cross-references and multi-hop reasoning | **PageIndex** — LLM can follow references |
| High-stakes domains (legal, finance, medical) | **PageIndex** — explainable, exact citations |
| Massive corpus, simple keyword/semantic lookup | **Vector search** — faster, cheaper at scale |
| One-time queries across many short documents | **Vector search** — index cost too high for PageIndex |
| Real-time ingestion of streaming documents | **Vector search** — lower latency indexing |

**Hybrid approach:** Use vector search to identify _which documents_ are
relevant, then use PageIndex for precise extraction _within_ those documents.

## 8. Trade-offs and Limitations

- **Indexing cost**: Building the tree is more expensive than chunking +
  embedding. Each node summary requires an LLM call. Best suited for documents
  that will be queried multiple times.
- **Latency**: Retrieval involves multiple LLM calls (tree navigation is
  iterative), making it slower than a single vector similarity lookup.
- **Input formats**: Currently PDF and Markdown only.
- **LLM dependency**: Both indexing and retrieval require an LLM (currently
  defaulting to GPT-4o). Quality and cost are tied to the underlying model.
- **Corpus-level search**: PageIndex excels at _within-document_ retrieval. For
  searching across thousands of documents, a pre-filtering step (vector search,
  metadata, or keyword) is still needed.
- **Open-source scope**: The public repo covers tree index generation. The full
  retrieval engine (LLM tree search, chat API) runs as a cloud service.

## 9. Key Concepts Glossary

| Term | Definition |
|---|---|
| **Tree Index** | Hierarchical JSON structure representing a document's logical organization (like a rich table of contents with summaries) |
| **Node** | A single element in the tree, mapping to a section of the document with title, page range, summary, and children |
| **LLM Tree Search** | Retrieval algorithm where an LLM iteratively navigates the tree by reasoning about which branches contain the answer |
| **Hybrid Tree Search** | Variant that combines LLM reasoning with other search signals |
| **Vision-based RAG** | Pipeline that retrieves page images instead of OCR text, using a vision-language model |
| **Reasoning-based RAG** | The overarching paradigm: retrieval via logical reasoning rather than embedding similarity |
| **Sufficiency check** | Step where the LLM evaluates whether gathered information fully answers the query before generating a response |
| **MCP** | Model Context Protocol — standard for exposing tools to LLM agents |
