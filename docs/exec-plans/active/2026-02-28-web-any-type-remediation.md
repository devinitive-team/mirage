# Eliminate Explicit `any` from Web TypeScript Sources

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows the requirements in `docs/PLANS.md` and must be maintained in accordance with that file.

## Purpose / Big Picture

After this change, the frontend TypeScript code under `web/src/` will no longer rely on explicit `any` for application-owned code, and the PDF highlight flow will use concrete library types end-to-end. A developer will be able to run a repeatable audit command and see no explicit `any` in authored TypeScript files, then run type and quality checks with no regressions.

The user-visible behavior remains the same: the preview page still renders a PDF, highlight selection still works, and existing upload/search workflows remain unchanged.

## Surprises & Discoveries

- Observation: Explicit `any` appears only in one authored source file and one generated file.
  Evidence: `rg` scan found `web/src/routes/preview.tsx` (5 explicit `any` spots) and `web/src/routeTree.gen.ts` (`as any` twice).
- Observation: The generated route tree is produced by TanStack Router and explicitly warns against manual edits.
  Evidence: Header in `web/src/routeTree.gen.ts` states it is auto-generated and should not be modified by hand.
- Observation: TanStack Router generator currently hardcodes `as any` for route updates when `disableTypes` is false.
  Evidence: `web/node_modules/@tanstack/router-generator/src/generator.ts` emits `} as any)` in `createUpdateRoutes`.
- Observation: `npx tsc --noEmit` already fails before this work because `PdfViewer` uses `unknown` and casts that do not match `PdfHighlighter` callback types.
  Evidence: TS2322 errors in `web/src/components/PdfViewer.tsx` for `highlightTransform` and highlight position typing.

## Context and Orientation

The web frontend lives in `web/src`. The explicit `any` remediation scope is centered on PDF highlight rendering and callback typing:

- `web/src/routes/preview.tsx` currently declares `highlightTransform` parameters with explicit `any`.
- `web/src/components/PdfViewer.tsx` has similar callback logic using `unknown` and type assertions, which currently fails standalone TypeScript checks.
- `web/src/routeTree.gen.ts` contains `as any` but is generated by TanStack Router and excluded from lint checks by `web/biome.json`.

`react-pdf-highlighter` ships type definitions in `web/node_modules/react-pdf-highlighter/dist/`. The callback contract for `PdfHighlighter` expects typed highlight, rectangle, and scaling functions. Using those exact exported types avoids drift and removes hand-wavy casts.

Quality commands are defined in `docs/QUALITY.md`: `task build`, `task format`, `task lint`, `task test`, plus interactive browser validation for UI/behavior changes.

## Plan of Work

Milestone 1 establishes typed primitives for PDF callbacks. In `web/src/routes/preview.tsx`, replace explicit `any` function parameters with precise types imported from `react-pdf-highlighter`. The `highlight` argument should be represented as a viewport highlight shape (`IHighlight` plus viewport `position`). The `setTip`, `viewportToScaled`, and `screenshot` callbacks should use library rectangle/content types so signatures align with `PdfHighlighter` expectations.

Milestone 2 applies the same typing model to `web/src/components/PdfViewer.tsx`, removing the existing `unknown` fallback and reducing `as IHighlight` casts. Keep behavior unchanged: clicking highlights still opens the tip, and `onSelectionFinished` still constructs new highlights correctly.

Milestone 3 adds prevention so explicit `any` does not reappear in authored code. Use one of two low-risk options: either configure Biome to make `noExplicitAny` an error for included files or add a dedicated audit command (for example `rg`-based) wired into `web` scripts/Taskfile. Preserve the existing generated-file exclusions (`routeTree.gen.ts`) and document why.

Milestone 4 validates with project quality commands and records evidence. For UI parity, run the required browser interaction flow from `docs/QUALITY.md` against the preview experience and confirm no behavior regressions.
